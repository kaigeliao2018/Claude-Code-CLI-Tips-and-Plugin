**Agent Teams启动方法**
* **环境配置方法**：可以在settings.json文件中添加对应配置，也可以在终端中通过export命令设置`cloud code experimental agent teams`环境变量，两种方式均可。
* **团队创建流程**：环境配置完成后，直接用自然语言告诉Cloud创建对应团队、指定任务和模型，Cloud就会自动完成团队搭建、任务分配和协调执行，整个过程不需要手动编写配置文件。

**Agent Teams底层架构**
* **架构类比说明**：整个架构可以类比真实办公室：team lead相当于项目经理坐在中间，每个teammate相当于独立工位的工程师，整体逻辑和真实开发团队一致。
* **共享协作机制**：共享看板对应shared task list，所有人都可以查看待办、领取任务、标记任务完成； teammate之间通过类似即时通讯的mailbox机制直接发消息沟通。
* **本地存储特点**：团队配置文件存储在本地的cloud teams目录下，任务列表存储在cloud tasks目录下，开发者可以直接打开文件查看团队状态，还可以手动编辑任务列表进行调试。

**运行状态查看的两种显示模式**
* **in process模式说明**：该模式下所有队友都在当前同一个终端中运行，通过shift加方向键可以切换查看不同队友的工作状态。它的优点是所有终端都能使用，缺点是同一时间只能查看一个队友的状态。
* **split pans模式说明**：该模式类似监控室的多屏幕墙，每个队友占用一个独立窗口，可以同时查看所有人的进度，点击对应窗口即可和对应队友沟通。它需要Tmax或者item 2的支持，系统会默认自动选择适配的模式，在Tmax环境中会默认使用该模式。

**Agent Teams控制操作与高阶功能**
* **基础控制操作**：可以要求队友先提交开发方案，经过审批后再动手开发；可以随时关闭某个指定队友的任务；任务全部完成后，可以让team lead执行清理团队操作完成统一收尾。
* **质量门禁高阶功能**：可以通过teammate idle hook和task completed hook设置自动检查，类似工厂流水线的质检环节，每完成一道工序就会自动跑一轮检查，不合格就会打回重做，以此保证最终产出的质量。

**官方总结的最佳实践**
* **上下文与团队规模要求**：给队友分配任务时必须提供充足的上下文，因为队友不会继承team lead的对话历史；团队规模需要控制在3-5人，人数过多会提升协调成本，更容易引发队友之间的冲突。
* **任务分配与冲突避免**：每个队友分配5-6个任务，任务太少会造成能力浪费，任务太多容易导致超负荷；每个队友负责不同的文件，从源头避免文件冲突，尽管系统自带文件锁，仍推荐从任务分配阶段就做好隔离。
* **启动阶段实践建议**：项目启动时先安排调研任务，让队友先读取代码、运行测试、熟悉开发环境，确认环境无误后再动手修改代码。

**当前版本的功能限制**
* **不支持嵌套团队**：当前版本不支持嵌套团队，不能让已有团队的某个队友再自行组建下级小团队。
* **会话与角色限制**：每个会话只能存在一个Agent团队，团队成员的角色一旦确定后就不能更换人员。
* **运行恢复限制**：在in process模式下，如果队友运行中断，无法恢复中断前的运行状态。

**核心心法总结与实操建议**
* **核心心法归纳**：Agent Teams的核心心法可以总结为四句话：给队友充足上下文，控制团队规模在3~5人，合理拆分任务力度，善用质量门禁，核心逻辑是像管理真实团队一样管理AI团队。
* **实操体验建议**：建议听众立刻打开终端，设置对应环境变量，用自然语言描述创建一个三人团队，亲自体验一次Agent Teams的功能。
