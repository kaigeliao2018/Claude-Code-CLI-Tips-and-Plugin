**Cloud Mind规则不生效的原因**
* **规则失效原因**：在Cloud Mind中书写的规则本质是建议，不是强制约束，AI在上下文拥挤、任务复杂时就会遗忘规则。
* **规则强制方案**：hooks是硬编码的拦截器，挂载在生命周期事件上，可以实现规则的强制执行，不需要依赖AI自觉执行。
* **hooks类比说明**：Cloud Mind中自定义规则相当于口头告诉孩子别碰插座，hooks相当于直接装上插座保护盖，实现物理阻断。

**hooks的支持事件与实用效果**
* **支持事件类型**：hooks一共支持五种生命周期事件，分别是pre to use、post to use、post to use failure notification、User prompt submit等。
* **最实用事件**：日常开发中最实用的是pre to use和post to use两种事件。
* **实际效率提升**：配置post to use hook后，每次编辑完文件会自动触发格式检查，不合规范立刻阻断编辑，一百次编辑累计可以节省1到2小时的回溯时间。

**prompt缓存与compaction机制**
* **缓存分层规则**：system prompt采用分层缓存策略，基础指令、工具定义做全局缓存，CLAUDE DMD按项目缓存，对话消息每轮递增。
* **compaction触发逻辑**：当对话长度增加、上下文窗口快满时，compaction会自动触发，将整段对话发给模型做摘要，压缩到约20000 TOKEN后替换旧消息。
* **成本控制效果**：虽然做整段对话摘要会产生调用成本，但因为system prompt和工具定义已经命中缓存，整体成本仅为未缓存的1/10，远低于预期。

**任务编排机制进化**
* **旧版本任务机制**：Cloud Code早期的任务编排是todos模式，属于单agent线性清单。
* **新版本任务机制**：目前Cloud Code已经进化到tasks模式，支持多agent并行执行任务，同时支持任务依赖关系配置与状态跟踪。
* **设计核心优势**：tasks模式是isolation surface设计的核心，可以实现上下文与权限的隔离，达成受控自治的效果。

**避免过度工程化的建议**
* **行业普遍现象**：AI开发领域中，初学者和真正的高手都会选择保持设计简洁，只有中间水平的开发者会堆砌功能。
* **常见堆砌行为**：常见的过度设计行为包括堆砌20个skills、编写1000行cloud code、创建100个subagents。
* **开发原则**：AI开发需要保持设计简洁，避免不必要的功能堆砌，避免过度工程化带来的问题。

**常见开发反模式与修复方案**
* **反模式1：知识过度塞入**：将所有知识都塞进prompt中，导致关键指令被稀释，修复方案为仅保留契约和禁止项，资料拆分到skills和rules中。
* **反模式2：skill功能混杂**：将知识库、部署脚本等多种功能放在同一个skill中，导致触发不稳定，修复方案为一个skill只负责一类功能。
* **反模式3：缺少验证闭环**：仅靠AI自行判断任务是否完成，缺少外部验证，修复方案为每类任务绑定一个verifier做验证。
* **反模式4：权限未清理**：settings.json中残留已批准的危险命令权限，触发后会造成不可逆的问题，修复方案为定期审查allowed tools列表，清理无用权限。

**AI开发者的三个成长阶段**
* **阶段1：工具使用者**：工具使用者只关注功能的使用方法，不涉及自定义配置。
* **阶段2：流程优化者**：流程优化者开始自主编写CLAUDEMD和自定义skills，优化自身工作流程。
* **阶段3：系统设计者**：系统设计者会思考如何让agent在约束下自主运作，设计人与AI的协作框架。

**AI开发能力质变的核心**
* **能力质变的本质**：AI开发能力的真正质变，不是学会更多的AI命令，而是掌握用系统思维设计人与AI的协作界面。
* **入门系统设计的方法**：可以从给最常出错的操作写一个pre-use hook开始，比如在每次文件编辑前自动检查项目规范。
* **入门效果**：从这一个hook开始实践，就踏进了AI系统设计者的门槛。
